

# 이번주 멘토링 회고

```
Q1. Statement와 preparedStatement의 차이를 말씀해주세요
Q2. JDBC에서 고민해보면 좋은 트랜잭션이라는 개념에 대해 말씀해주세요
Q3. 이후 멘토링 과정에서 ORM에 대해서 얘기를 할텐데, 이제는 사실 JPA를 쓸텐데 JDBC를 왜 굳이 알아야 할까요?
Q4. 커넥션 풀의 개수는 개발자가 설정할 수 있을까요 없을까요?
Q5. 개발자가 커넥션풀의 개수를 어떤 기준으로 정해야할까요? 
Q6. HikariCP는 그냥 라이브러리라고 생각하면 되겠네요?
Q7. 클러스터형 인덱스는 몇개까지 생성 가능할까요?
Q8. 인덱스가 들어가는 컬럼은 어떤 특징이 있으면 좋을까요?
```

> ##### [학습이 필요한 부분에 대해 직접적인 문제 상황 제시와 함께 해답을 찾도록 유도해주심]
```
Q1. JDBC를 이용해 DB의 users라는 테이블의 이름, 나이, 성별을 받아와 user라는 자바 객체에 옮기는 과정을 말씀해주세요
Q2. 요청에 따라 입력받은 고객의 정보를 가져오는 쿼리는 Statement가 좋을까요 preparedStatement가 좋을까요?
Q3. 개발자가 커넥션풀의 개수를 어떤 기준으로 정해야할까요?  
    - 실제 쿼리 요청 주기와 쿼리 수행 시간을 예시로 '동시에 처리할 수 있는 커넥션'의 개수로 커넥션 풀을 예상할 수 있음을 인지시켜주심
Q4. 커넥션 풀의 개수가 쓰레드 풀의 개수보다 많으면 어떤 상황이 발생할까요?
    - 실제 커넥션 풀과 쓰레드 풀의 관계를 예시로 들어주시며 커넥션 풀과 쓰레드 풀의 관계에 대해 인지시켜주심
Q5. 2개 이상의 애플리케이션에서 각각 10개의 커넥션 풀을 받으면 실제 DB에 연결된 커넥션 풀은 몇개일까요?
    - 2개 이상의 애플리케이션에서 같은 DB의 커넥션 풀을 받을 때 각 애플리케이션 담당자들이 간과할 수 있는 부분에 대해 인지 시켜주심
    - 이런 문제를 해결하는 방법으로 간혹 '통합'된 툴을 사용하는 경우가 있음
Q6. 사용자 테이블이란 걸 만들 때 클러스터드 인덱스는 어떤 컬럼에 넣어주는게 좋을까요?
Q7. 새로운 사용자가 생성되면 ID는 어떻게 관리하는게 좋을까요?
Q8. ID가 랜덤으로 올라간다면 어떤 문제가 생길 수 있나요?
    - 클러스터형 인덱스는 물리적인 정렬이 필요하므로 데이터 삽입시 비교적 비싼 비용을 지불해야 할 수 있음을 인지 시켜주심.
Q9. 사용자가 1000명 있었는데, 중간에 사용자들이 많이 삭제가 되는 경우 어떤 일이 일어날까요
    - 디스크 단편화를 방지하기 위해 특정 DB에서는 인덱스 유지 비용이 무시할 수 없는 수준으로 일어날 수 있음을 인지 시켜주심
    - 이 방법에 대한 해결 중 하나로 'isDelete'라는 컬럼 추가를 통해 가상으로 삭제하는 방법을 제시해 주시면서 비슷한 문제 해결에 '메모리와 작업 속도 사이의 트레이드 오프 관계'를 고려해 볼 수 있도록 유도해주심.
    - 이런 개념을 'hard delete'와 'soft delete'라고 표현함을 인지시켜주심
Q10. 사용자 테이블을 화면에 출력토록 데이터를 클라이언트로 내려줘야 하는데 고객이 '모든 컬럼에 정렬, 역순 정렬, 필터링'을 해야 한다고 했을 때 여기서 논 클러스터링 인덱스를 어떻게 가져가실건가요?
    - 수정, 삽입시에 기술적인 문제가 분명히 있음을 인지시켜주고 고객에게 어떤 식으로 제안할지에 대한 셋업을 해주심
Q11. 비즈니스 요구자를 말한 고객님에게 어떤 질문이나 추천을 고려하실 수 있으실까요?
    - 사용자 수가 적을 때, 사용자 수가 30000명 일 때, 30000명인데 사용자 수가 계속 증감될 때, 클라이언트에게 정렬 역할을 넘길 수 있을 때, 실제 이 기능의 필요성에 대해 검증이 되지 않았을 때
Q12. 코드를 짜는 법만 아는 초보 개발자한텐 어떻게 설명해야 이해할 수 있을까요?
    - 직접 코드를 통해 보여주는 게 가장 쉽게 이해될 수 있음을 인지시켜주셨다.
Q13. 찬혁님이 코드를 짜신 뒤에 내가 SOLID를 잘 지키고 계신지 어떻게 알 수 있을까요?
    - 코드를 짜고 나서 내가 솔리드를 위반했는지 일일이 보는 것 보다는 '하나의 원칙만 가지고 계속 코딩을 해서 체화 시켜보는' 방법을 적용하면 굳이 신경 쓰지 않아도 솔리드를 준수하게 될 수 있다는 팁을 알려주셨다.
```

> ###### [좋았던 점]
```
* 예전에 왜 이게 나왔는지를 보면 지금 상태를 알 수 있기에 과거를 보는 건 굉장히 좋습니다.
* 개발자란게 단순히 개발 하는 능력만 중요한게 아니라 요구사항을 명확히 파악하고 어떤 기술을 적용할 지에 대한 고민을 할 수 있는 능력이 굉장히 중요하구나 라는 사실을 깨달았다.
* 확실한 기본기가 없고 고민할 수 있는 능력이 없으면 요구사항을 전달하는 고객에게 '요구사항을 제한하는' 해결책 밖에 제시하지 못함을 깨달았다.
* 비즈니스 요구사항이 신빙성이 있지 않더라도 팀 내부 동료들의 의심과 무시를 뚫고 나에게 요구사항을 가져왔을 고객 혹은 기획자를 도우려면 데이터 분석이나 그 요구사항의 근거를 같이 도출해 내는 방법도 고민해보면 좋을 거 같다.
```

> ###### [보완하고 싶은 점]
```
* 무조건 실습 코드 다 써보기로 마음 먹었는데, 커넥션 풀이나 인덱스 관련 내용에 대해선 코드 한 줄 작성치 못했다. 
* 주제 자체에 집중해야 했는데 '트랜잭션 물어보셨으니까 격리 레벨이랑 ACID 물어보시겠지 ..' 라던지 '클러스터형 인덱스면 B트리 물어보실거고, 대수 확장성이 어쩌고 ..'  같은 고민들을 했는데 말 그대로 주제파악을 못했다. 이번 주 부터는 딴 곳에 눈 돌리지 말고 본 주제에만 집중해보기로 결심했다.
* 이번주는 학습 하는 과정에 끈기가 부족했던 거 같다. 커넥션 풀 관련해서도 실제 예시 코드를 작성해보고, 최적화 방법들엔 어떤것들이 있는지 봤어야 했는데 질문 양에 집착해서 적당히 하고 넘어간 감이 있었다. 이 부분 좀 더 고민해보려 한다.
* 말을 어눌하게 하는 감이 없지않아 있는 거 같다. 신경써서 고쳐봐야겠다.
```
